# Context Snapshot

- Task statement: Create a website where WCA competitors can check which U.S. states they have competed in, based on WCA weekly results exports collected by a cron job.
- Desired outcome: An execution-ready specification for a public WCA state-coverage website with personal lookup and leaderboard features.
- Stated solution: Scheduled export ingestion + derived state coverage data + leaderboard of top N competitors by number of U.S. states visited + WCA ID search showing visited/unvisited states, last 3 competitions per state, and upcoming competitions when present.
- Probable intent hypothesis: Turn raw WCA export data into a compelling, easy-to-browse geography-centric stats product for competitors, with both personal utility and public discovery value.
- Known facts/evidence:
  - Repo is brownfield: existing TanStack Start app with routes under `src/routes`.
  - `package.json` shows TanStack Start, React 19, Mantine, Drizzle, PostgreSQL, Wrangler, and Cloudflare tooling.
  - `src/db/schema.ts` already models `competitors`, `regions`, and `competitor_regions`, suggesting prior work toward region/state tracking.
  - `src/routes/index.tsx` is still starter-template content, so the product UI is not implemented yet.
  - `src/routes/__root.tsx` currently includes a `Region Tracker` button, which aligns with the domain but appears placeholder-level.
  - No visible cron/ingestion pipeline or WCA-specific UI routes are implemented in the inspected files.
- Constraints:
  - No implementation during deep-interview mode.
  - Need explicit non-goals and decision boundaries before handoff.
  - Brownfield questions should use repository evidence when possible.
- Unknowns/open questions:
  - Primary success target for V1: personal lookup, leaderboard engagement, trip planning, or all of the above.
  - What exactly counts as a visited state from the export data.
  - Whether the site is U.S.-only in V1 or should preserve a path to countries/regions later.
  - Whether unauthenticated public use is sufficient.
  - Freshness and failure expectations for the scheduled export sync.
  - Exact meaning of "top N" and who chooses N.
  - How upcoming competitions should be sourced and displayed.
- Decision-boundary unknowns:
  - Whether OMX may choose UI layout, route structure, DB normalization, caching strategy, and leaderboard defaults without confirmation.
  - Which product decisions require explicit user approval.
- Likely codebase touchpoints:
  - `src/routes/__root.tsx`
  - `src/routes/index.tsx`
  - new route files under `src/routes/`
  - `src/db/schema.ts` and migrations
  - API/server routes under `src/routes/api*`
  - deployment/runtime config for scheduled ingestion
